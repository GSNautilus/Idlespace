## How to work on this project (read this — it matters)

The user values **small, test-gated increments** and got burned earlier this branch
by a big-bang change that "comprehensively did not work" and had to be reverted.
The working loop that's been succeeding:

1. Make ONE coherent, small change (a "slice").
2. Syntax-check it (see below).
3. Hand it to the user with a concrete **test plan**; they run the game in a browser and confirm.
4. **Only then commit.** Never commit unverified work. Never bundle multiple slices.
5. If a slice is big, split it (e.g. "stop creating X" then "delete dead X code" as two commits).

Other hard-won rules:
- **Don't guess — look.** Read the actual code before asserting. Several wasted cycles
  came from proposing fixes without reading. When the user reports a bug, read the
  relevant function before theorizing.
- **The user tests in the browser themselves.** You can't. So "syntax passes" is a weak
  signal — describe exactly what to look for and wait for confirmation.
- **Ask on genuine forks.** Control schemes, entity-model shape, tuning targets — when a
  decision is the user's to make, use AskUserQuestion rather than assume.
- The user runs a local server with `python -m http.server 8765` from the project root
  and tests in **Firefox** (no Chrome — never use Chrome tooling). Hard-refresh
  (Ctrl+Shift+R) needed to bypass cache. Note: earlier a "bug" was actually the python
  server having crashed — if behavior seems frozen, suspect the server.

## Why this rework exists

v0.2.0's "Combat 2.0" was originally attempted as one giant change layered on top of the
**old idle combat system** (fleets lock together, a red line connects them, they trade
volleys with no player input). That failed badly and was reverted. The user then
redirected: **abandon the old idle combat entirely and rebuild as an RTS.**

Design decisions the user locked in (via Q&A):
1. **Ships are individual units.** No "fleet" entity. A "fleet" is just a group of ships
   the player moves together (like RTS units + selection).
2. **Group moves match the slowest ship's speed and stay together** (cohesive formation).
3. **No hard fleet-size cap.** The admiral's aura is the only thing that scales with grouping.
4. **Box-select** for grouping. Control scheme: left-drag box-selects, left-click selects a
   unit / moves the selection on empty space, **right-drag pans**, right-click deselects.
5. Combat must respect **sensor honesty** — nothing engages the player unless the player is
   within that ship's sensor range, and vice-versa.
6. Combat happens **on the starmap** (no combat overlay). A future subtle notification +
   persistent combat-reports panel is wanted (see remaining work), NOT a focus-stealing modal.

## The new model — technical reference (current line numbers ~, verify before editing)

### Entities
- `mapEntities[]` holds **only** `type:"ship"` entities now. Shape:
  `{ type:"ship", owner:"player"|"npc", card, admiral|null, faction?, stance, x, y, destX, destY, speed, sensorRange, stealth, label, _moveSpeed? }`.
- `card` carries per-instance combat state: `_combatShields`, `_combatArmor`, `_dead`,
  plus `_shotCooldownLeft`. These round-trip via `CARD_INSTANCE_FIELDS`.
- `deployShipGroup(cards, opts)` [~3524] is the one spawn helper. opts: `{owner, stance, faction, admiral, x, y}`.

### Combat — `tickShipCombat(dt)` [~3871], called from the game loop
- For each alive ship: nearest enemy within `min(sensorRange*50, maxWeaponRange)`.
- Fires every in-range weapon when `_shotCooldownLeft` hits 0; cooldown = `getShipFireInterval(card)` [~7006].
- `calcHitChance` [~3685]: base 0.40 + sensor − stealth − speed. Misses render flying past the target.
- `applyDamageByWeapon(card, type, dmg)` [~3788]: COMBAT_TRIANGLE multipliers, shields→armor, sets `_dead`.
- Admiral aura: `collectAdmiralAuras()` [~3580] once per tick; `auraBonusAt()` gives best `commandBonus%`; firer inside aura deals `dmg * (1+bonus/100)`.
- Per-kill loot via `rollPerKillLoot` [~3816] + `applyLoot`. Dead entities removed at end of tick (then `onEntityRemoved`).
- Key combat constants [~7005–7018]: `BASE_FIRE_INTERVAL=1.5`, `COMBAT_HP_SCALE=30`, `RANGE_THRESHOLDS={long:1000,medium:650,short:350}`, `COMBAT_TRIANGLE`.
- `getEntityShips(ent)` [~3863] returns `[ent.card]` and `getShipWorldPos(ent)` [~3554] returns `{x:ent.x,y:ent.y}` — trivial now, kept so the combat loop stays index-based and a future sub-ship model can slot back in.

### Selection & movement
- `selectedEntities` (Set of entity refs) [~3473]; `panelEntity` (single entity whose detail panel is open).
- Helpers: `clearSelection`, `selectOnly(ent)`, `selectedPlayerUnits()`, `onEntityRemoved(ent)` [~3495] (call after any splice).
- Box-select: `boxSelect`/`boxSelectStart` state; rectangle drawn at end of `render()`; `finalizeBoxSelect()` [~6669] selects player ships whose screen pos is inside.
- `issueGroupMove(movers, tx, ty)` [~6697]: translate whole group by (target − centroid), set each `ent._moveSpeed = min(speed)` so they arrive together. `tickEntities` uses `_moveSpeed ?? speed`, clears on arrival.
- Pointer handlers: pointerdown gates pan (right/middle/touch) vs box-select (left); `handlePointerEnd` finalizes box / handles click. contextmenu = deselect.
- Panels: `showEntityPanel(ent)` [~6256] single; `showGroupPanel()` [~6340] multi (currently count + rarity breakdown + All-Aggressive/Evasive — **needs the full card roster, see below**).

### Admiral
- Rides aboard a ship: `ent.admiral`. Aura radius `getAdmiralAuraRadius` [~3572] = `200 + maxFleetSize*12`.
- Gold ring + faint gold aura circle in `drawEntities`. Permadeath = dies with ship; disband returns to colony queue.
- `getMaxFleetShips` [~3556] still exists and is still used by the **deploy staging UI** (`refreshFleetTab` [~8818]) to cap how many ships you stage at once — vestigial, slated for removal in Slice 4.

### Slice 4 — selection & deploy polish (next)
1. **Multi-select panel shows all ships' cards** *(user-requested, top priority)*. `showGroupPanel`
   currently shows only a count + rarity breakdown. It should render a scrollable roster of every

2. **Deploy rework.** Drop the vestigial `getMaxFleetShips` staging cap in `refreshFleetTab`;
   relabel "Deploy Fleet"/"Deploy Solo" (misnomers now) to a single "Deploy"; optionally let the
   player choose which ship carries the admiral instead of auto-assigning the rarest.
3. **Selection visual tweaks** — "N selected" HUD, cleaner rings.
4. **(minor) Group-move formation** — optionally snap to a tidy formation on arrival.

### Slice 5 — combat behavior (bigger; confirm scope with user first)
Ships currently **hold position and fire** — they don't move on their own in combat, and
**stances do nothing differential**. Candidate work (this is the original task tree's
"Combat behavior / Ship movement / Ship targeting", reframed for RTS):
- Aggressive ships auto-close on detected enemies; evasive try to keep distance / disengage.
- Range optimization: brawlers close to short range, snipers hold at max weapon range.
- Per-ship target priority (closest/weakest/strongest) and/or manual focus-fire.
- A subtle combat-occurring notification (red `!` indicator, non-focus-stealing) + a persistent,
  dismissable combat-reports panel so off-screen fights are reported. Multiple simultaneous
  combats must not overwrite each other. (User specified this earlier.)
The user may prefer to keep "hold-and-fire" + manual micro and skip some of this. **Ask.**
